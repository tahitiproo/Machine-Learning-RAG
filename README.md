# Machine-Learning-RAG

Презентация: "Advanced Hybrid RAG: Архитектура многостадийного поиска"

Слайд 1: Титульный
Заголовок: Высокоточная RAG-система на базе гибридного поиска и Cross-Encoder реранкинга.
Подзаголовок: Сочетание семантической близости, ключевых слов и глубокого анализа релевантности.
Визуал: Схема "Воронка": 1 000 000 документов → 250 кандидатов → 30 для реранкинга → Топ-5 для LLM.

Слайд 2: Проблема и решение (The Challenge)
Проблема: Обычный векторный поиск (Naive RAG) теряет точность на сложных запросах, не учитывает специфические термины и "галлюцинирует" при выборе контекста.
Решение в коде:
Двухстадийный поиск (Retrieval + Reranking).
Гибридная схема (Dense + Sparse).
Улучшенное чанкование с сохранением метаданных (Titles/H1).

Слайд 3: Общая архитектура системы
Компоненты:
Ingestion: Чанкование с перекрытием (Overlap) и дедупликацией.
Retrieval: Параллельный поиск в FAISS (E5) и BM25.
Fusion: Объединение результатов через RRF (Reciprocal Rank Fusion).
Refinement: Расширение запроса через PRF (Pseudo Relevance Feedback).
Reranking: Финальная проверка через BGE-Reranker-v2-m3.

Слайд 4: Подготовка данных (Ingestion Deep Dive)
Стратегия: Paragraphs → Sentences → Word-based Windows (160 words).
Инновация: Приклеивание иерархии заголовков (Title — H1) к каждому чанку.
Зачем это нужно: Семантический вектор чанка "Как настроить роутер" без заголовка может быть слабым. С заголовком "Инструкция к модели X" он становится уникальным.

Слайд 7: Sparse Search: BM25 и PRF
Зачем нам BM25? Векторный поиск (E5) часто ошибается в артикулах, цифрах и редких именах. BM25 находит их по точному совпадению.
PRF (Pseudo Relevance Feedback):
Делаем пробный поиск.
Берем ключевые слова из топ-результатов.
Обогащаем исходный запрос.
Результат: Сокращение "Query Drift" (ухода запроса от темы).

Слайд 9: Устройство Cross-Encoder (BGE-Reranker-v2-M3)
Архитектура: Full Interaction Transformer.
Процесс: В модель подается пара [CLS] Query [SEP] Passage.
Внутренняя логика: В отличие от E5, здесь токены запроса "видят" токены документа на каждом слое через Self-Attention.
Выход: Один лоджит релевантности.

Слайд 10: Bi-Encoder vs Cross-Encoder: Архитектурная битва
Bi-Encoder (E5): Похож на поиск книги в библиотеке по шифру на обложке (быстро, но поверхностно).
Cross-Encoder (BGE): Похож на прочтение каждой страницы каждой книги экспертом (медленно, но максимально точно).
В нашей системе: Мы используем Bi-Encoder для поиска кандидатов, а Cross-Encoder — как эксперта для финального выбора.

Слайд 11: Сравнение реранкеров: BGE vs Cohere vs Jina
BGE-M3: State-of-the-art для RU и мультиязычности. Поддерживает длинный контекст.
Cohere Rerank: Очень мощный, но проприетарный.
Jina Reranker: Хорош для очень длинных документов, но требует больше ресурсов.
Наш выбор: BGE-v2-M3 — баланс между скоростью на GPU и качеством понимания русского языка.

Слайд 12: Оптимизация производительности (Latency)
Проблема: Cross-Encoder медленный.
Как мы это решили:
Реранкинг только Топ-30 документов.
Агрегация чанков (Top-2 Max-pooling) перед реранкером.
Использование torch.inference_mode() и GPU-батчинга.

Слайд 13: Итоги и метрики качества
Система устойчива к:
Синонимам (благодаря E5).
Специфическим терминам (благодаря BM25).
Шуму в вебе (благодаря дедупликации и реранкингу).
Что дальше? Внедрение LLM (Llama 3 / GigaChat) для генерации финального ответа на основе отобранного контекста.

Слайд 14: Заключение (Q&A)
Key Takeaway: Эффективный RAG — это не просто векторная база данных, а многослойный фильтр информации.
